35. Backend orientado por contratos: como evitar regras de negócio espalhadas

Um agente adicionou uma regra nova para criação de pedidos: nenhum pedido poderia ser criado com valor total abaixo de um centavo. A regra foi escrita direto dentro do handler da rota HTTP, junto com a validação de formato e a chamada ao banco de dados. Três semanas depois, outro agente implementou um segundo ponto de entrada para criar pedidos — um endpoint de importação em lote, usado por um sistema parceiro — e replicou a lógica de acesso ao banco, mas não encontrou a regra do valor mínimo, porque ela nunca tinha saído de dentro da rota original. Pedidos de valor zero passaram a ser criados silenciosamente pela importação em lote.

A regra de negócio existia. Só existia no lugar errado: presa a um único ponto de entrada, em vez de estar disponível para qualquer código que precisasse criar um pedido.

Rota fina, regra centralizada

Rotas finas e contratos explícitos evitam acoplamento e duplicação.

Um backend saudável separa quatro responsabilidades que frequentemente aparecem misturadas no mesmo arquivo: transporte (receber a requisição HTTP), aplicação (orquestrar o caso de uso), domínio (as regras de negócio em si) e persistência (salvar e recuperar dados). Quando a regra de valor mínimo vive dentro do transporte — a rota — ela só se aplica a quem passa por aquela rota específica. Qualquer outro caminho de entrada para o mesmo domínio simplesmente não a herda.

Separando o que estava misturado

// Antes: tudo misturado na rota
app.post("/orders", async (req, res) => {
  if (!req.body.items) return res.status(400).send("invalid");
  if (req.body.total_cents < 1) return res.status(400).send("invalid total");
  const order = await db.orders.create(req.body);
  res.json(order);
});

// Depois: regra de domínio isolada e reutilizável
function validarPedido(pedido: PedidoInput) {
  if (pedido.total_cents < 1) {
    throw new RegraDeNegocioError("Valor total deve ser maior que zero.");
  }
}

// A rota vira fina, delegando ao domínio
app.post("/orders", async (req, res) => {
  const pedido = parseOrderInput(req.body);
  validarPedido(pedido); // mesma função usada por QUALQUER entrada
  const order = await orderService.criar(pedido);
  res.json(order);
});

Com validarPedido isolada do transporte, o endpoint de importação em lote — ou qualquer outro ponto de entrada futuro — pode reutilizar exatamente a mesma regra, em vez de precisar redescobri-la e reimplementá-la (ou esquecê-la, como aconteceu).

O contrato como fronteira entre camadas

A separação entre camadas só funciona se cada uma se comunicar com a seguinte através de um contrato explícito — os tipos de entrada e saída de cada função — e não através de acesso direto a estruturas internas de outra camada. É a mesma lógica dos contratos técnicos entre sistemas diferentes, aplicada agora dentro das fronteiras internas do próprio backend.

O limite: nem todo endpoint precisa de quatro camadas separadas

Para um serviço pequeno, com uma única rota e nenhuma perspectiva realista de outro ponto de entrada, separar rigidamente as quatro camadas pode ser engenharia excessiva. A separação compensa quando existe — ou é plausível que venha a existir — mais de um caminho de entrada para a mesma regra de negócio.

Conclusão

Uma regra de negócio presa dentro de uma rota HTTP não é uma regra de negócio protegida — é uma regra que só vale para quem passa por aquele caminho específico. Extrair a regra para uma camada de domínio isolada, com contrato próprio, é o que garante que ela seja aplicada de forma consistente, não importa quantos pontos de entrada o sistema venha a ter.
