33. Mocks como contrato temporário entre frontend e backend

Para não esperar o backend ficar pronto, um agente de frontend criou dados simulados para a tela de pedidos: datas como objetos Date do JavaScript, valores monetários como números de ponto flutuante, status como texto livre em português. Quando o endpoint real finalmente ficou pronto, nada batia: datas chegavam como strings ISO, valores vinham em centavos como inteiro, status eram códigos em inglês. A integração, que deveria ser trivial, virou uma reescrita significativa da camada de exibição — porque o mock nunca tinha sido desenhado como uma representação do contrato real, apenas como algo conveniente para preencher a tela naquele momento.

O mock cumpriu a função de destravar o desenvolvimento paralelo. Não cumpriu a função de representar fielmente o que o frontend precisaria consumir depois — e ninguém tinha um plano para essa transição.

Mock útil representa contrato, não conveniência

Mocks úteis representam contratos reais e possuem plano de substituição.

Um mock existe para permitir que duas partes de um sistema avancem em paralelo sem esperar uma pela outra — o mesmo problema discutido no artigo sobre planejamento orientado por dependências. Mas essa vantagem só se mantém quando o mock representa fielmente o contrato acordado, não uma aproximação inventada por conveniência. Um mock que diverge do contrato real não elimina a dependência entre frontend e backend — apenas adia a colisão para o momento da integração.

Construindo o mock a partir do contrato, não da imaginação

// contrato acordado (ver artigo 18 desta série)
interface Order {
  id: string;
  total_amount_cents: number;
  status: "pending" | "confirmed" | "cancelled";
  created_at: string; // ISO 8601
}

// mock.ts — gerado a partir do MESMO contrato
export const mockOrder: Order = {
  id: "ord_123",
  total_amount_cents: 15000,
  status: "confirmed",
  created_at: "2026-03-10T14:22:00Z",
};

Construído dessa forma, o mock não é uma invenção paralela — é uma instância concreta do mesmo contrato que o backend vai implementar. Quando o endpoint real ficar pronto, a camada de exibição já espera exatamente o formato que vai receber.

O plano de substituição que faltava

Além de representar o contrato corretamente, um mock útil precisa de um plano explícito para deixar de existir:

  • Onde ele vive: isolado num único ponto do código, fácil de localizar e remover — não espalhado por múltiplos componentes.
  • Quando ele sai: uma condição clara, como "remover assim que o endpoint /api/orders estiver disponível em homologação".
  • Quem verifica a remoção: um item de checklist de pull request, não a memória de quem criou o mock.

O limite: mocks nem sempre precisam desse rigor

Para um protótipo descartável, feito para validar uma ideia em uma demonstração e sem intenção de virar produção, exigir que o mock siga um contrato formal é esforço desnecessário. O rigor compensa quando o mock alimenta desenvolvimento paralelo que vai, de fato, integrar com um sistema real depois.

Conclusão

Um mock não é apenas um atalho para não esperar o backend. É um contrato temporário — e, como qualquer contrato, só é confiável quando representa fielmente o que vai substituí-lo e quando existe um plano claro para essa substituição acontecer. Sem isso, o mock resolve o problema de curto prazo e cria um problema maior de integração para depois.

O mock que seu frontend usa hoje foi desenhado a partir do contrato real com o backend, ou foi inventado por conveniência para preencher a tela?
