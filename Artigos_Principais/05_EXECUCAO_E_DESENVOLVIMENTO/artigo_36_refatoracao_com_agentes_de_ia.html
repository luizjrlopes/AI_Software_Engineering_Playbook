36. Refatoração com agentes de IA: como melhorar o código sem mudar seu comportamento

A tarefa parecia inofensiva: "limpar a função de cálculo de preços, está difícil de ler". O agente reorganizou a lógica, extraiu variáveis com nomes melhores e simplificou uma cadeia de condicionais aninhadas. O código ficou, de fato, mais legível. Também mudou, de forma sutil, a ordem em que um arredondamento era aplicado num caso específico de desconto combinado — suficiente para alterar o resultado em centavos, num cenário raro que os poucos testes existentes não cobriam. A mudança só foi percebida semanas depois, quando alguém notou uma pequena divergência em faturas específicas.

O agente cumpriu literalmente o que foi pedido — "limpar" o código — mas ninguém tinha definido, antes da tarefa começar, o que exatamente deveria permanecer idêntico depois da limpeza.

Refatorar exige saber o que não pode mudar

Refatorar exige caracterizar o comportamento antes de alterar a estrutura.

Refatoração, por definição, é mudar a estrutura interna de um código sem alterar seu comportamento observável. Essa definição só é verificável se alguém souber, de antemão, qual é o comportamento observável completo — incluindo os casos raros que ninguém tem em mente no momento de "limpar o código". Sem essa caracterização prévia, refatoração e mudança de comportamento ficam indistinguíveis até alguém notar a diferença em produção.

Testes de caracterização: fotografar o comportamento antes de mexer

Um teste de caracterização não verifica se o comportamento está "correto" — verifica exatamente o que o código faz hoje, incluindo casos que ninguém documentou formalmente, para servir de referência depois da mudança estrutural: 

// Antes de refatorar, capturar o comportamento atual
describe("calcularPreco - caracterização", () => {
  it("preço base sem desconto", () => {
    expect(calcularPreco(10000, [])).toBe(10000);
  });

  it("desconto percentual isolado", () => {
    expect(calcularPreco(10000, [{ tipo: "percentual", valor: 10 }]))
      .toBe(9000);
  });

  it("desconto combinado com frete grátis (caso raro)", () => {
    // Este resultado é o comportamento ATUAL, correto ou não —
    // serve de referência para a refatoração.
    expect(
      calcularPreco(10000, [
        { tipo: "percentual", valor: 10 },
        { tipo: "frete_gratis" },
      ])
    ).toBe(9000);
  });
});

Com esses testes rodando antes e depois da refatoração, qualquer mudança de resultado — mesmo em centavos, mesmo num caso raro — aparece imediatamente como uma falha de teste, em vez de ser descoberta semanas depois numa fatura de cliente.

O fluxo correto de uma refatoração protegida

  1. Escrever testes de caracterização cobrindo o comportamento atual, incluindo casos raros.
  2. Confirmar que os testes passam antes de qualquer mudança estrutural.
  3. Refatorar a estrutura interna.
  4. Rodar os mesmos testes novamente — qualquer falha indica mudança de comportamento, intencional ou não.

O limite: nem toda refatoração precisa desse rigor completo

Para uma função pequena, isolada, sem histórico de bugs e sem uso em cenários financeiros ou críticos, escrever testes de caracterização formais antes de uma limpeza simples pode ser desproporcional. O rigor compensa quando a função lida com dinheiro, dados sensíveis ou lógica cujo comportamento exato já causou problema antes.

Conclusão

"Deixar o código mais limpo" e "mudar o que o sistema calcula" são coisas diferentes, mas fáceis de confundir sem uma referência explícita do comportamento anterior. Testes de caracterização são essa referência — não provam que o código estava certo antes, apenas garantem que a refatoração não mudou nada além da estrutura.

A última "limpeza de código" que um agente fez no seu projeto tinha algum teste capturando o comportamento anterior, ou dependeu de alguém perceber a diferença depois?
