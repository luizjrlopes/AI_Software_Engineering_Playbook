26. Agente de testes: como fazer a IA provar que o código funciona

O mesmo agente que implementou o fluxo de login escreveu também os testes automatizados para ele. Os testes passavam, todos verdes. Só cobriam, porém, o cenário em que o e-mail e a senha estavam corretos — exatamente o caminho que o próprio agente já tinha em mente quando escreveu o código. Nenhum teste tentava uma senha errada três vezes seguidas, nenhum tentava um e-mail inexistente, nenhum verificava o que acontecia se o banco de dados demorasse para responder. Os testes não avaliaram o código: eles confirmaram o que o autor já acreditava que tinha implementado certo.

Isso não é uma falha de honestidade do agente. É uma tendência natural de quem escreve testes para o próprio código: testar aquilo que se sabe que funciona é mais fácil — e mais confortável — do que procurar ativamente o que pode estar quebrado.

Teste existe para encontrar falha, não para confirmar sucesso

O objetivo do teste é encontrar falhas, não confirmar o implementador.

Um agente de testes, atuando com esse objetivo declarado, parte de uma postura diferente da de quem implementou a funcionalidade: em vez de perguntar "isso funciona?", pergunta "de que forma isso poderia falhar, e o código já está preparado para essa falha?". Essa mudança de pergunta é o que separa um teste que confirma da autora e um teste que investiga de forma independente.

Três categorias de caso que um teste investigativo cobre

  • Casos positivos: o comportamento esperado quando tudo está correto — a base mínima, não o teto do que testar.
  • Casos negativos: entradas inválidas, credenciais erradas, permissões insuficientes — o que deveria ser rejeitado com uma resposta clara, não com uma falha silenciosa.
  • Casos de borda: condições extremas ou raras — um campo vazio, um valor máximo, uma resposta lenta do servidor, tentativas repetidas em sequência.

Aplicando as três categorias ao login

describe("login", () => {
  it("autentica com e-mail e senha corretos", () => { /* positivo */ });

  it("rejeita senha incorreta sem revelar se o e-mail existe", () => {
    /* negativo */
  });

  it("bloqueia após 5 tentativas falhas consecutivas", () => {
    /* borda */
  });

  it("não trava a interface se o serviço de auth demorar
      para responder", () => {
    /* borda */
  });
});

Nenhum desses quatro testes existia na versão original, porque nenhum deles fazia parte do "caminho que já se sabia que funcionava". Um agente de testes, com o mandato explícito de procurar falhas, é quem preenche essa lacuna — de preferência, sem ser a mesma instância que escreveu a implementação original.

O limite: nem todo código exige a mesma profundidade de teste

Cobrir exaustivamente casos de borda para um script interno descartável, usado uma única vez, é esforço sem retorno proporcional. A profundidade de teste deveria acompanhar o risco: autenticação, pagamento e qualquer lógica que manipula dados sensíveis justificam essa investigação completa; ferramentas internas de uso pontual, normalmente não.

Conclusão

Um agente de testes que compartilha os mesmos pressupostos de quem implementou o código tende a testar apenas o que já acredita estar certo. Definir o teste como uma busca ativa por falhas — não como uma confirmação de sucesso — é o que transforma uma suíte de testes verdes em evidência real, e não em uma cerimônia que apenas repete a confiança do implementador.

Os testes do seu último recurso foram escritos para confirmar que o código funciona, ou para tentar provar que ele falha?
