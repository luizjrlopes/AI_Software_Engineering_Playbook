29. Skills para agentes de IA: capacidades reutilizáveis além dos prompts

Toda vez que alguém precisava de uma auditoria de segurança numa API, a tarefa começava do zero: uma pessoa escrevia um prompt longo explicando o que verificar, outra pessoa, semanas depois, escrevia um prompt diferente para a mesma finalidade, cobrindo alguns pontos e esquecendo outros. O resultado da auditoria variava dependendo de quem tinha escrito o prompt daquele dia — não porque as APIs fossem diferentes, mas porque o procedimento de auditoria nunca tinha sido definido uma única vez, de forma reutilizável.

O conhecimento de "como auditar uma API corretamente" existia, mas distribuído em prompts individuais, cada um uma reconstrução parcial e inconsistente do anterior.

Skill é procedimento reutilizável, não prompt bonito

Uma skill reúne instrução, ferramenta, conhecimento, entradas, saídas e critérios de sucesso.

Uma skill, no contexto de agentes de IA, é mais do que um prompt bem escrito guardado num arquivo. É uma capacidade documentada de forma completa o suficiente para ser executada de maneira consistente por qualquer agente, em qualquer momento, sem depender de quem lembrou de incluir qual detalhe naquele dia específico.

Os seis elementos de uma skill completa

  • Instrução: o procedimento passo a passo a ser seguido.
  • Ferramentas: o que a execução dessa skill pode acionar — scanners, linters, chamadas de API.
  • Conhecimento: referências e critérios técnicos necessários para executar com precisão.
  • Entradas: o que precisa ser fornecido para a skill funcionar.
  • Saídas: o formato esperado do resultado.
  • Critérios de sucesso: como saber se a execução foi completa e correta.

A skill de auditoria de API, documentada uma única vez

skill: auditoria_de_api_rest

entradas:
  - especificacao_openapi_ou_codigo_fonte
  - lista_de_endpoints_criticos (opcional)

instrucao:
  1. Verificar autenticação obrigatória em todos os endpoints
     que alteram estado.
  2. Verificar limite de taxa (rate limiting) em endpoints
     públicos.
  3. Verificar validação de entrada em todos os parâmetros.
  4. Verificar que mensagens de erro não vazam detalhes
     internos (stack trace, versão de biblioteca).
  5. Verificar logging de acesso sem dados sensíveis em texto
     puro.

saida:
  - lista de achados no formato:
    { endpoint, problema, severidade, recomendacao }

criterio_de_sucesso:
  - todos os endpoints da especificação foram avaliados
  - cada achado tem severidade e recomendação preenchidas

Com essa skill documentada uma única vez, qualquer agente que a execute — hoje ou daqui a seis meses — produz uma auditoria com a mesma cobertura e o mesmo formato, independentemente de quem disparou a tarefa.

O limite: nem todo procedimento vale a pena virar skill

Documentar uma skill completa para uma tarefa executada uma única vez, sem expectativa de repetição, é esforço desproporcional. A prática compensa para procedimentos recorrentes — auditorias, revisões padronizadas, processos de deploy — não para tarefas pontuais e não repetíveis.

Conclusão

Um prompt bem escrito resolve uma execução. Uma skill documentada resolve a execução, a próxima, e a seguinte, com a mesma cobertura e o mesmo padrão de qualidade — sem depender da memória de quem escreveu o prompt daquele dia específico.
