44. Qualidade além dos testes: acessibilidade, desempenho, segurança e experiência do usuário

A feature passou em todos os testes automatizados, com cobertura acima de 90%. No dia do lançamento, quatro problemas diferentes apareceram, nenhum deles capturado por um único teste verde: a tela levava oito segundos para carregar com o volume real de dados de produção, o contraste dos botões não atendia aos critérios mínimos de acessibilidade, uma dependência com vulnerabilidade conhecida estava em uso havia semanas, e usuários reais simplesmente não entendiam o que um dos botões fazia. Nenhum teste funcional falhou, porque nenhum deles tinha sido desenhado para verificar essas quatro dimensões.

Testes funcionais provam que o código faz o que foi programado para fazer. Não provam que fazer isso é suficiente para o software ser considerado bom.

Funcionar é apenas uma das dimensões de qualidade

Software pode funcionar e ainda ser inseguro, lento, inacessível ou confuso.

Qualidade de software é multidimensional. Testes automatizados cobrem principalmente a dimensão funcional — o sistema faz o que deveria fazer no nível lógico. Mas desempenho, acessibilidade, segurança e experiência do usuário são dimensões independentes, cada uma com seus próprios critérios de avaliação, que raramente aparecem numa suíte de testes de unidade convencional.

Quatro dimensões, quatro perguntas diferentes

  • Desempenho: o sistema responde dentro de um tempo aceitável com volume de dados real, não apenas com os poucos registros de um ambiente de teste?
  • Acessibilidade: pessoas que dependem de leitor de tela, navegação por teclado ou alto contraste conseguem usar a funcionalidade?
  • Segurança: as dependências utilizadas estão livres de vulnerabilidades conhecidas, e os dados sensíveis estão protegidos?
  • Experiência do usuário: uma pessoa sem conhecimento prévio do sistema entende o que cada elemento da interface faz?

Avaliando a mesma feature nas quatro dimensões

## Checklist multidimensional — feature de exportação de relatório

- [ ] Desempenho: testado com volume real de produção
      (10.000+ registros), resposta em menos de 3s
- [ ] Acessibilidade: botão de exportação possui nome
      acessível; contraste validado (WCAG AA)
- [ ] Segurança: dependências verificadas contra
      vulnerabilidades conhecidas (scan executado)
- [ ] Experiência do usuário: testado com pelo menos um
      usuário sem contexto prévio da funcionalidade

Nenhum item desse checklist é substituível por um teste unitário verde — cada um exige uma verificação própria, com ferramenta ou processo específico para aquela dimensão.

O limite: nem toda feature exige as quatro dimensões completas

Uma ferramenta interna, de uso esporádico por uma única pessoa da equipe técnica, pode dispensar auditoria de acessibilidade completa. O investimento em cada dimensão deveria ser proporcional ao alcance e à criticidade da funcionalidade — recursos voltados a milhares de usuários externos justificam rigor nas quatro dimensões; ferramentas internas de nicho, normalmente não.

Conclusão

Um teste automatizado verde é uma prova estreita: prova que a lógica funciona, não que o software é seguro, rápido, acessível ou compreensível. Tratar qualidade como sinônimo de "testes passando" deixa de fora exatamente os problemas que mais afetam quem realmente usa o sistema.

A última feature que passou em todos os testes no seu projeto já foi avaliada em desempenho, acessibilidade, segurança e experiência do usuário, ou só na dimensão funcional?
