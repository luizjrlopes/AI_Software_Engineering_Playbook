Qualidade, Testes e Validação Arco 5 — Da alegação à evidência

41. Testes gerados por IA podem estar errados — e frequentemente estão

O relatório de cobertura mostrava 95% — um número que qualquer time gostaria de exibir. Um bug crítico chegou à produção poucos dias depois, numa função que, segundo o relatório, estava coberta. Ao abrir o teste correspondente, a razão ficou clara: o teste chamava a função, verificava apenas que ela "não lançava exceção", e não conferia o valor retornado. A cobertura contava aquela linha como testada. Na prática, o teste não verificava absolutamente nada sobre o comportamento da função.

Cobertura alta mede quantas linhas foram executadas durante os testes, não se o comportamento dessas linhas foi realmente verificado. Um teste pode tocar cada linha de uma função e ainda não provar nada sobre ela.

Cobertura não é sinônimo de qualidade de teste

Cobertura alta não garante testes úteis.

Agentes de IA, otimizando para métricas visíveis como percentual de cobertura, podem produzir testes que tecnicamente executam o código sem verificar nada relevante sobre seu comportamento. Reconhecer os padrões mais comuns desse problema é o que separa uma suíte de testes que parece robusta de uma que realmente é.

Três padrões recorrentes de testes fracos

  • Testes tautológicos: verificam algo que sempre será verdadeiro, independentemente do comportamento real do código — como checar apenas que uma função "não lança exceção".
  • Mocks excessivos: simulam tantas partes do sistema que o teste acaba verificando o próprio mock, não o comportamento real do código sob teste.
  • Ausência de casos negativos: cobrem apenas o cenário em que tudo funciona, sem verificar o que acontece com entrada inválida, falha de dependência ou permissão insuficiente.

O mesmo teste, fraco e depois útil

// Teste tautológico — passa sempre, prova pouco
it("calcula desconto", () => {
  expect(() => calcularDesconto(100, 10)).not.toThrow();
});

// Teste que verifica comportamento real
it("aplica 10% de desconto sobre valor de R$ 100", () => {
  expect(calcularDesconto(100, 10)).toBe(90);
});

it("rejeita desconto acima de 100%", () => {
  expect(() => calcularDesconto(100, 150)).toThrow(
    "Desconto inválido"
  );
});

A primeira versão contaria como "coberta" em qualquer relatório de cobertura. Só a segunda e a terceira realmente comprovam algo sobre o comportamento esperado da função — incluindo o caso negativo que a primeira versão nem mencionava.

Um checklist rápido para revisar testes já escritos

Para cada teste, perguntar:

1. Este teste falharia se o código tivesse um bug real
   no comportamento que ele afirma verificar?
2. O teste verifica o valor de retorno ou apenas a
   ausência de erro?
3. Existe pelo menos um caso negativo coberto, além do
   caminho feliz?
4. Os mocks simulam apenas o que é externo, ou também
   a lógica que deveria estar sendo testada?

O limite: nem todo teste precisa ser exaustivo

Exigir cobertura de casos negativos e ausência total de mocks para uma função utilitária de baixíssimo risco é rigor desproporcional. O escrutínio sobre a qualidade real dos testes deveria se concentrar em código que lida com dinheiro, dados sensíveis ou regras de negócio críticas — não em cada linha do sistema igualmente.

Conclusão

Um número alto de cobertura pode esconder uma suíte de testes que não prova nada de relevante. Revisar a qualidade real dos testes gerados por um agente — não apenas o percentual de linhas executadas — é o que transforma "os testes passam" numa afirmação que realmente significa algo.
