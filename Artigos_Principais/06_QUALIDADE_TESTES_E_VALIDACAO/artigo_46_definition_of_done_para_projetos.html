46. Definition of Done para projetos desenvolvidos com agentes de IA

O projeto tinha uma Definition of Done — um parágrafo numa página da wiki interna, escrito havia mais de um ano, dizendo genericamente que "o código deve estar testado e documentado antes de ir para produção". Duas equipes diferentes, trabalhando com agentes de IA no mesmo projeto, terminavam tarefas com níveis de rigor completamente diferentes: uma exigia testes de caso negativo e relatório de execução reproduzível; a outra aceitava "os testes passaram" como suficiente. Ambas diziam estar seguindo a mesma DoD, porque o parágrafo da wiki era vago o bastante para justificar qualquer uma das duas interpretações.

Uma Definition of Done que não é operacionalizada como checklist verificável não alinha nada — vira um texto que cada equipe lê como quiser.

DoD é o padrão mínimo, não uma aspiração

DoD alinha qualidade mínima entre tarefas, pessoas e agentes.

Este artigo já tratou, no início da série, a definição de pronto para uma tarefa individual, cobrindo quatro dimensões: funcional, técnico, documental e operacional. Uma Definition of Done de projeto amplia essa mesma lógica para o padrão que vale para toda tarefa que passa por ali — reunindo, num único checklist verificável, tudo o que este bloco discutiu: gates que bloqueiam avanço sem validação, evidências reproduzíveis em vez de alegações, testes que provam algo real, critérios de aceite executáveis e achados de auditoria específicos.

Uma DoD de projeto, operacionalizada

## Definition of Done — projeto Checkout

Uma tarefa está pronta quando:

### Código
- [ ] Lint e verificação de tipos sem erros
- [ ] Nenhuma duplicação óbvia de lógica já existente

### Testes
- [ ] Casos positivos, negativos e de borda cobertos
- [ ] Relatório de execução reproduzível anexado
      (comando, ambiente, saída completa)

### Documentação
- [ ] Mudanças de contrato de API refletidas na
      documentação correspondente

### Segurança
- [ ] Nenhuma dependência nova com vulnerabilidade
      conhecida de severidade alta ou crítica

### Operação
- [ ] Log ou métrica que permite observar a mudança
      após o deploy

Com esse checklist explícito, as duas equipes do início deste artigo não teriam mais espaço para interpretações divergentes: a DoD deixa de ser um parágrafo de intenção e passa a ser uma lista binária, verificável por qualquer pessoa — ou qualquer agente — antes de considerar uma tarefa concluída.

DoD como contrato entre humanos e agentes

Uma DoD operacionalizada dessa forma cumpre a mesma função para agentes de IA que cumpre para desenvolvedores humanos: remove a ambiguidade sobre o que "pronto" significa naquele projeto específico, independentemente de quem — ou o que — executou a tarefa. Ela deveria fazer parte do Context Pack de qualquer tarefa delegada a um agente, não existir apenas como documentação de referência raramente consultada.

O limite: DoD única para todo tipo de tarefa é rígida demais

Aplicar a mesma DoD completa, com todas as verificações, a uma correção trivial de texto na interface é desproporcional. Uma DoD madura permite níveis diferentes de rigor conforme o risco da tarefa — mas isso precisa ser uma variação explícita e documentada, não uma interpretação livre de cada equipe sobre o que a DoD "realmente" exige.

Conclusão

Uma Definition of Done vaga produz o mesmo problema, em escala de projeto, que uma Issue vaga produz em escala de tarefa: cada pessoa, e cada agente, preenche a ambiguidade com seu próprio critério. Transformar a DoD num checklist verificável e explícito é o que garante que "pronto" signifique a mesma coisa, não importa quem tenha feito o trabalho.

A Definition of Done do seu projeto hoje é um checklist verificável, ou um parágrafo vago o suficiente para significar coisas diferentes para equipes diferentes?
