54. Segurança por padrão em projetos desenvolvidos com agentes de IA

Ninguém pediu explicitamente para não fazer aquilo — e por isso mesmo aconteceu. Um agente, criando um microsserviço novo do zero, sem partir de nenhum template existente, configurou CORS aceitando qualquer origem, registrou o corpo completo das requisições em log — incluindo senhas enviadas em texto puro durante testes — e colou uma chave de API diretamente no código-fonte, porque isso funcionava e ninguém tinha proibido explicitamente. Cada uma dessas decisões, tomada de forma independente, parecia inofensiva no momento. Juntas, formaram um serviço inseguro por omissão, não por má intenção de ninguém.

O agente não violou nenhuma regra, porque nenhuma regra de segurança estava presente por padrão no ambiente em que ele trabalhou. A ausência de proibição funcionou, mais uma vez, como permissão implícita.

Segurança não pode depender de lembrança

Segurança não deve depender de o agente lembrar de aplicá-la.

Esperar que um agente "lembre" de aplicar boas práticas de segurança a cada tarefa é repetir o mesmo erro discutido em outros artigos desta série sobre contexto e escopo: informação que depende de memória, humana ou artificial, eventualmente falha. Segurança por padrão significa que as práticas seguras já estão embutidas no ponto de partida — no template, na configuração base, na infraestrutura — de forma que o caminho mais fácil para o agente seja também o caminho seguro.

Quatro princípios mínimos embutidos por padrão

  • Menor privilégio: o serviço só acessa exatamente os recursos de que precisa, nada além disso, por configuração — não por disciplina de quem escreve o código.
  • Validação de entrada: todo dado externo é validado antes de ser processado, por um middleware padrão do template, não por decisão caso a caso.
  • Proteção de dados sensíveis: campos como senha e token nunca aparecem em log, porque o sistema de logging já os filtra automaticamente.
  • Gestão de segredos: chaves de API e credenciais vêm de um cofre de segredos, nunca de uma variável hardcoded no código.

O mesmo serviço, partindo de um template seguro

// template-service/config.ts (parte do scaffold padrão)
export const corsConfig = {
  origin: ALLOWED_ORIGINS, // lista explícita, nunca "*"
};

export const logger = createLogger({
  redact: ["password", "token", "authorization"], // filtro automático
});

export const secrets = loadFromVault(); // nunca hardcoded

Quando o agente parte desse template em vez de começar do zero, replicar os três problemas do início do artigo exigiria uma decisão ativa de contornar a configuração segura — não apenas a ausência de uma decisão de proteger.

O limite: segurança por padrão não substitui revisão

Um template seguro reduz a chance de erros comuns, mas não elimina a necessidade de revisão para riscos específicos do domínio — uma regra de negócio sensível ainda pode ser implementada de forma insegura mesmo dentro de um template bem configurado. Segurança por padrão é uma rede de proteção contra os erros mais previsíveis, não uma garantia completa.

Conclusão

Um agente de IA, na ausência de restrição explícita, tende a escolher o caminho estatisticamente mais comum — que nem sempre é o mais seguro. Embutir segurança no ponto de partida, em vez de esperar que ela seja lembrada a cada tarefa, é o que garante que o caminho mais fácil também seja o caminho seguro.
