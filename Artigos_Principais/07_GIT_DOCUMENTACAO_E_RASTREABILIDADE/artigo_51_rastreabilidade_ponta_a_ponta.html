Git, Documentação e Rastreabilidade Arco 6 — Da alteração à rastreabilidade

51. Rastreabilidade ponta a ponta: do requisito ao commit, teste e deploy

Durante um incidente, alguém precisava responder duas perguntas simples: por que esse trecho de código existe, e ele foi validado antes de ir para produção? A Issue original tinha sido fechada sem nenhuma referência ao pull request que a implementou. O pull request, por sua vez, tinha sido mesclado sem menção clara a qual commit específico continha a mudança relevante. E o log de deploy mostrava apenas "versão 2.4.1 implantada", sem indicar quais mudanças específicas aquela versão continha. Cada elo da cadeia existia isoladamente. Nenhum se conectava ao seguinte.

A informação necessária para responder às duas perguntas estava, tecnicamente, espalhada pelo sistema. Só não estava conectada de um jeito que permitisse segui-la em sequência, do requisito original até a versão em produção.

Rastreabilidade é uma corrente, não pontos isolados

Rastreabilidade permite explicar por que algo existe e como foi validado.

Este bloco já tratou, separadamente, boas práticas para commits, pull requests e documentação. Rastreabilidade ponta a ponta é o que conecta esses artefatos numa cadeia contínua: cada elo referenciando o anterior, de forma que seguir do requisito ao deploy seja uma questão de navegação, não de investigação arqueológica.

A cadeia completa, elo por elo

Issue #482 "Corrigir cálculo de desconto combinado"
    ↓ referenciada em
Commit a1b2c3d "fix(checkout): corrige desconto (#482)"
    ↓ incluído em
Pull Request #97 "Corrige desconto combinado com frete grátis"
    ↓ com testes anexados: discount.test.ts (3 passed)
    ↓ mesclado e taggeado como
Release v2.4.1
    ↓ registrado no changelog
CHANGELOG.md — v2.4.1: "Corrige #482 (desconto combinado)"

Com essa cadeia intacta, a investigação do incidente levaria minutos: a versão em produção referencia o changelog, o changelog referencia a Issue, a Issue referencia o commit e o pull request, e o pull request contém a prova de que o cenário específico foi testado antes do deploy.

Onde a cadeia costuma quebrar

  • Issue sem referência ao commit: perde-se a ligação entre intenção e implementação.
  • Pull request sem prova anexada: perde-se a evidência de validação, mesmo que a mudança esteja correta.
  • Deploy sem changelog vinculado: perde-se a capacidade de saber o que uma versão específica contém, sem investigar manualmente.

O limite: rastreabilidade completa tem custo de manutenção

Manter essa cadeia intacta para cada mudança, incluindo ajustes triviais, consome tempo de documentação desproporcional ao risco. A cadeia completa compensa para mudanças que afetam comportamento observável do sistema — a mesma fronteira discutida em outros artigos deste bloco — não para cada commit isolado do repositório.

Conclusão

Rastreabilidade não é sobre ter cada artefato individualmente bem documentado — é sobre eles se conectarem em sequência, permitindo que qualquer pessoa siga do requisito original até a versão em produção sem depender de memória ou investigação manual. Quando essa cadeia se rompe em qualquer elo, a rastreabilidade deixa de existir na prática, mesmo que cada peça isolada pareça bem cuidada.
