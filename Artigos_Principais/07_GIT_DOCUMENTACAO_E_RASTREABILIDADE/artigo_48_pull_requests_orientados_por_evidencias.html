48. Pull requests orientados por evidências para código gerado por IA

O pull request tinha apenas um título — "fix bug" — e o diff. Nenhuma descrição do problema original, nenhuma explicação do porquê daquela abordagem, nenhuma menção a testes, nenhuma indicação do que fazer se algo desse errado depois do deploy. O revisor, sob pressão de prazo, leu o código, achou razoável e aprovou. Dias depois, quando um efeito colateral apareceu em produção, ninguém conseguia responder rapidamente: essa mudança tinha sido testada para esse cenário? Havia um plano de reversão conhecido?

O código em si podia estar correto. O pull request, como documento de decisão, não continha nada que permitisse avaliar isso com confiança — só convidava à aprovação por confiança no tom, não por evidência.

Um pull request é um documento de decisão, não só um diff

Uma PR deve explicar intenção, mudança, provas, riscos e rollback.

Um pull request bem estruturado permite que um revisor avalie uma mudança em minutos, com confiança, porque a informação necessária já está ali — não precisa ser reconstruída através de perguntas no chat ou de leitura extensa de código sem contexto. Isso vale igualmente para mudanças feitas por humanos e por agentes de IA: a estrutura do PR é o que carrega o contexto da decisão, não apenas o resultado dela.

Os cinco elementos de um PR revisável

  • Intenção: qual problema esta mudança resolve, ou qual Issue ela atende.
  • Mudança: um resumo do que foi alterado, em linguagem natural, complementando o diff.
  • Provas: evidência de que a mudança funciona — saída de testes, captura de tela, resultado de migração.
  • Riscos: o que pode dar errado, ou o que exige atenção específica na revisão.
  • Rollback: como reverter a mudança, caso necessário depois do deploy.

Um template aplicado ao mesmo bug

## Intenção
Corrige Issue #491: erro 500 ao aplicar cupom expirado
no checkout.

## Mudança
Adiciona validação de data de expiração antes de aplicar
o cupom; retorna erro 400 com mensagem clara em vez de 500.

## Provas
$ npm test -- coupon.test.ts
✓ rejeita cupom expirado com erro 400
✓ aceita cupom válido normalmente
2 passed, 0 failed

## Riscos
Nenhuma migração de dados. Sem impacto em cupons já
aplicados anteriormente.

## Rollback
Reverter este commit isoladamente; nenhuma dependência
de outras mudanças no mesmo deploy.

Com essa estrutura, o revisor do início deste artigo teria a resposta às duas perguntas que ficaram sem resposta depois do incidente: sim, havia teste cobrindo o cenário de cupom expirado, e sim, o rollback seria simples e isolado.

O limite: nem todo PR precisa dos cinco elementos completos

Uma correção trivial e de baixíssimo risco não precisa de uma seção de rollback elaborada. O rigor do template deveria ser proporcional ao impacto da mudança — mudanças em fluxos críticos justificam os cinco elementos completos; ajustes cosméticos podem se contentar com intenção e mudança.

Conclusão

Um pull request que só contém um diff transfere para o revisor todo o trabalho de reconstruir intenção, risco e plano de reversão — trabalho que quem abriu o PR já tinha essa informação em mãos. Estruturar essa informação de forma explícita é o que transforma revisão em avaliação informada, em vez de aprovação por confiança no tom da descrição.

O último pull request que você aprovou continha prova de teste e plano de rollback, ou apenas um diff e a confiança de que provavelmente estava certo?
