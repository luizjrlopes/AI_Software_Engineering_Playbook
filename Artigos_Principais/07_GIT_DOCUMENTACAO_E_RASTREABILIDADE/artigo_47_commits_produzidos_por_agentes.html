Git, Documentação e Rastreabilidade Arco 6 — Da alteração à rastreabilidade

47. Commits produzidos por agentes de IA: como manter um histórico compreensível

Investigando um bug em produção, um desenvolvedor rodou git blame na linha suspeita e chegou a um commit com a mensagem "updates". O commit alterava quinze arquivos diferentes: parte de uma feature nova, uma correção não relacionada, e um ajuste de configuração. Nenhuma referência a uma Issue, nenhuma explicação do porquê daquela mudança específica. O histórico do Git existia, tecnicamente — mas não contava nada sobre o que tinha acontecido ali, nem por quê.

Um commit sem contexto não é apenas um detalhe estético do histórico. É uma lacuna de rastreabilidade que aparece exatamente no momento mais caro para descobri-la: durante uma investigação sob pressão.

O histórico do Git é documentação, não apenas mecanismo de versionamento

Histórico Git é parte da documentação operacional.

Cada commit é uma oportunidade de registrar não apenas o que mudou, mas por quê — e essa informação, acumulada ao longo do tempo, forma um histórico que qualquer pessoa, ou qualquer agente, pode consultar para entender a evolução do sistema. Tratar o commit como um detalhe administrativo, gerado automaticamente com mensagens genéricas, transforma esse histórico potencialmente valioso em ruído.

Da mudança grande aos commits atômicos

A mesma mudança de quinze arquivos, dividida em commits atômicos e vinculados à Issue original, ficaria assim: 

a1b2c3d feat(checkout): adiciona validação de cupom (#482)
e4f5g6h fix(checkout): corrige cálculo de frete grátis (#479)
h7i8j9k chore(config): atualiza timeout de conexão com gateway (#485)

Cada linha, sozinha, já responde três perguntas: o que mudou (o tipo e o escopo), qual a intenção (a descrição curta) e por que essa mudança existe (a referência à Issue). Um git blame nessas condições leva diretamente ao contexto da mudança, não a um mistério rotulado "updates".

Um padrão mínimo de mensagem de commit

<tipo>(<escopo>): <descrição curta e específica> (#<issue>)

tipo: feat, fix, refactor, chore, docs, test
escopo: o módulo ou área afetada
descrição: o que mudou, em uma frase, no imperativo
issue: referência obrigatória para mudanças não triviais

Esse padrão não exige ferramentas sofisticadas — exige apenas que cada commit, gerado por um agente ou por uma pessoa, siga uma estrutura mínima que preserva contexto para quem for consultar aquele histórico no futuro.

O limite: nem todo commit precisa de referência formal

Exigir referência a uma Issue para um commit de correção de digitação num comentário é rigor desproporcional. O padrão compensa para mudanças que afetam comportamento observável do sistema — a mesma distinção já discutida no artigo sobre código orientado por evidências.

Conclusão

Um agente de IA pode gerar commits com a mesma facilidade que gera qualquer outro artefato — a questão é se esses commits carregam contexto suficiente para servir de documentação no futuro, ou se são apenas marcações técnicas de "algo mudou aqui". Um histórico de commits bem escrito é, muitas vezes, a primeira e mais rápida fonte de verdade sobre por que o sistema é do jeito que é.

Se você rodasse git blame na linha mais crítica do seu sistema hoje, o commit encontrado explicaria o porquê da mudança, ou diria apenas "updates"?
