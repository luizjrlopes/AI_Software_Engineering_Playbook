Git, Documentação e Rastreabilidade Arco 6 — Da alteração à rastreabilidade

49. Documentação viva: como impedir que a IA atualize o código e abandone os documentos

Um agente adicionou um campo obrigatório novo ao contrato de uma API — document_id, exigido a partir daquela versão. O código mudou, os testes passaram, o pull request foi aprovado. A página de documentação da API, consultada por um time parceiro que integrava com esse endpoint, continuou descrevendo o contrato antigo, sem menção ao campo novo. Três semanas depois, a integração desse time parceiro começou a falhar silenciosamente, porque eles seguiram fielmente uma documentação que já não correspondia ao comportamento real do sistema.

Ninguém decidiu conscientemente deixar a documentação desatualizada. Simplesmente não havia nenhum mecanismo que conectasse a mudança de contrato à obrigação de atualizar aquela página específica.

Documentação confiável precisa de responsável e gatilho

Documentação confiável precisa de dono, gatilhos e validação.

Documentação que depende de alguém lembrar espontaneamente de atualizá-la tende a divergir do código com o tempo — não por negligência, mas porque atualizar documentação raramente parece urgente no momento em que o código já está funcionando. Documentação viva é aquela que tem um dono responsável por sua precisão, gatilhos explícitos que indicam quando ela precisa mudar, e alguma forma de validação que impede a divergência de passar despercebida.

Três elementos que mantêm a documentação sincronizada

  • Dono: uma pessoa ou time responsável por manter aquela documentação específica precisa — não "todo mundo", que na prática significa ninguém.
  • Gatilhos: uma lista explícita de que tipos de mudança exigem atualização de qual documento.
  • Validação: um mecanismo — humano ou automatizado — que verifica se o gatilho foi realmente atendido antes do merge.

Conectando mudança de contrato à atualização obrigatória

## Gatilhos de documentação — API de pedidos

| Tipo de mudança                     | Documento a atualizar         |
|--------------------------------------|--------------------------------|
| Novo campo obrigatório na resposta   | docs/api/orders.md             |
| Novo código de erro                  | docs/api/orders.md (seção erros) |
| Mudança de regra de negócio visível  | docs/domain/orders-rules.md    |

## Gate de PR
- [ ] Este PR altera o contrato da API de pedidos?
      Se sim, docs/api/orders.md foi atualizado?

Com esse gatilho explícito e um item de checklist correspondente no template de pull request — o mesmo tipo de gate discutido anteriormente neste bloco — a adição do campo document_id não passaria pela revisão sem a atualização correspondente da documentação.

O limite: nem toda documentação precisa de gatilho automatizado

Criar um gatilho formal para cada página de documentação interna, incluindo anotações de baixo impacto, é esforço desproporcional. Gatilhos explícitos compensam para documentação consultada por outras equipes ou sistemas externos — onde a divergência custa caro e é descoberta tarde — não para notas internas de uso pessoal.

Conclusão

Um agente de IA atualiza exatamente o que foi instruído a atualizar. Se a instrução cobre apenas o código, a documentação fica para trás — não por falha do agente, mas porque ninguém conectou a mudança de contrato à obrigação de manter a documentação correspondente em dia. Documentação viva exige esse vínculo explícito, não apenas boa vontade.
