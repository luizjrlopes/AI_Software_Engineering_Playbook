52. O README não é contexto suficiente para um agente de IA

O README do projeto era exemplar: instruções claras de instalação, comandos para rodar localmente, descrição resumida do que o sistema fazia. Um agente, recebendo apenas esse arquivo como contexto, implementou uma nova integração de pagamento chamando diretamente uma API externa a partir da camada de rotas — uma decisão que violava uma escolha arquitetural registrada meses antes, num ADR que dizia explicitamente que toda integração externa deveria passar por uma camada de adaptador dedicada, justamente para permitir trocar de provedor sem reescrever o domínio. O README nunca mencionava essa regra, porque nunca foi seu trabalho mencionar isso.

O agente não tinha como saber da regra. O README, por bem escrito que fosse, simplesmente não é o lugar onde esse tipo de decisão vive.

README orienta a entrada, não substitui o resto

README orienta entrada; não substitui arquitetura, decisões, operação e regras locais.

Um README cumpre bem uma função específica: ajudar alguém a instalar, rodar e entender superficialmente um projeto pela primeira vez. Tratá-lo como a fonte única de contexto para um agente de IA — a prática mais comum quando ninguém pensou explicitamente sobre isso — deixa de fora exatamente o tipo de informação que evita decisões erradas: arquitetura, decisões já tomadas, práticas operacionais e regras locais específicas do domínio.

O que existe além do README

  • Arquitetura: o mapa de módulos e responsabilidades discutido anteriormente neste bloco.
  • Decisões: os ADRs que registram por que certas escolhas foram feitas e não deveriam ser revertidas sem novo contexto.
  • Operação: como o sistema é implantado, monitorado e o que fazer em caso de incidente.
  • Regras locais: convenções e restrições específicas de um módulo, que não se aplicam ao projeto inteiro.

Um índice mínimo que o README poderia apontar

## docs/README.md — índice de documentação

- Arquitetura e módulos: docs/architecture/map.md
- Decisões arquiteturais: docs/adr/
- Runbook de operação: docs/ops/runbook.md
- Regras do domínio de pagamento: docs/domain/payments.md

Para tarefas que envolvem integração externa, consultar
OBRIGATORIAMENTE docs/adr/012-adapter-pattern.md antes de
implementar.

Esse índice não substitui o README — complementa ele, apontando explicitamente para onde vive o tipo de informação que evitaria o erro do exemplo inicial. Um Context Pack bem montado para uma tarefa de integração externa incluiria essa referência diretamente, sem depender de o agente adivinhar que ela existe.

O limite: nem todo projeto precisa da documentação completa

Um projeto pequeno, de escopo estável e sem decisões arquiteturais complexas, pode se sustentar razoavelmente bem com um README detalhado e pouco mais. A necessidade de documentação estruturada em múltiplas camadas cresce com a complexidade do domínio e o número de decisões acumuladas ao longo do tempo — não é um requisito universal desde o primeiro commit.

Conclusão

Um README responde "como eu rodo isso". Não responde "por que essa decisão foi tomada" ou "que regra este módulo específico precisa respeitar". Tratar o README como contexto suficiente para um agente de IA é assumir que ele sabe algo que nunca foi escrito em lugar nenhum que ele pudesse consultar.

Se um agente recebesse hoje apenas o README do seu projeto como contexto, ele saberia das decisões arquiteturais que não podem ser violadas?
