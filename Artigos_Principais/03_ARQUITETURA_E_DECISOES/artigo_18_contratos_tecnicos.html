18. Contratos técnicos: o que permite que vários agentes trabalhem no mesmo sistema

Um agente recebeu a tarefa de "deixar o schema de usuário mais claro" e, como parte disso, renomeou o campo full_name para name diretamente na resposta da API, sem tocar em nenhum outro serviço. A mudança fazia sentido isolada. O problema é que um aplicativo mobile, um job de exportação de relatórios e a interface web dependiam todos do nome antigo do campo. Nenhum deles quebrou no momento do deploy — o campo simplesmente parou de existir, e cada consumidor passou a exibir dados vazios silenciosamente, até alguém notar dias depois.

A tarefa não previa esse impacto porque ninguém tratou aquele campo como parte de um contrato compartilhado. Para quem escreveu a tarefa, era só um nome de propriedade num JSON. Para os outros três sistemas, era uma promessa que deixou de ser cumprida sem aviso.

Contrato é o que permite trabalho independente e seguro

Contratos estáveis desacoplam equipes e agentes.

Um contrato técnico — uma API, um schema de dados, uma definição de tipos ou um formato de evento — é o que permite que duas partes de um sistema evoluam de forma independente, desde que ambas continuem respeitando o que foi combinado. Quando múltiplos agentes trabalham no mesmo sistema, sem se coordenarem em tempo real, o contrato é o único ponto de verdade compartilhado entre eles. Alterá-lo sem aviso equivale a mudar as regras do jogo para quem já está jogando.

O mesmo schema, implementado por dois lados

A prática mais simples e mais eficaz é declarar o contrato antes da implementação, num formato que ambos os lados possam consultar — não descobrir por tentativa e erro: 

// contracts/user.schema.ts
export interface UserResponse {
  id: string;
  full_name: string;   // nome de exibição completo
  email: string;
  created_at: string;  // ISO 8601
}

Um agente de frontend e um agente de backend implementando contra essa mesma definição — versionada no repositório, não inventada de memória por cada um — produzem peças que se encaixam sem precisar de coordenação informal a cada mudança.

Evoluir contrato sem quebrar quem já depende dele

A forma correta de "deixar o schema mais claro" não é renomear o campo existente, e sim adicionar o novo nome ao lado do antigo, com um período de transição documentado: 

export interface UserResponse {
  id: string;
  full_name: string;   // @deprecated use `name` — remover em v3
  name: string;         // novo campo, substitui full_name
  email: string;
  created_at: string;
}

Essa abordagem custa mais no curto prazo — dois campos temporariamente redundantes — mas evita que consumidores desconhecidos do contrato quebrem silenciosamente. A remoção do campo antigo se torna uma etapa deliberada e comunicada, não um efeito colateral de uma tarefa de "limpeza".

O limite: nem toda interface interna precisa desse rigor

Aplicar esse nível de disciplina a uma função auxiliar usada apenas dentro de um único módulo, por um único agente, numa única tarefa, é rigor desnecessário. Contratos formais e versionamento cuidadoso valem a pena nas fronteiras realmente cruzadas por múltiplos consumidores — outros serviços, outros times, outros agentes trabalhando em paralelo — não em cada função privada do sistema.

Conclusão

Um contrato técnico não é apenas uma formalidade de documentação. É o mecanismo que permite que agentes e equipes diferentes construam partes distintas do mesmo sistema sem precisar reconfirmar, a cada tarefa, o que o outro lado espera receber. Tratar um schema compartilhado como detalhe de implementação é o que transforma uma limpeza de código em um incidente silencioso em produção.

Antes da última mudança de schema no seu sistema, alguém verificou quem mais dependia daquele campo além de quem escreveu a tarefa?
