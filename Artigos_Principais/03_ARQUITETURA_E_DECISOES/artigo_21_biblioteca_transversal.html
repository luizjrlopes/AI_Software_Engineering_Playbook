21. Biblioteca transversal: como centralizar boas práticas sem contaminar os projetos

A instrução parecia simples: "siga as diretrizes de engenharia da empresa" ao construir uma pequena API interna de conciliação financeira. O agente interpretou isso ao pé da letra e aplicou o conjunto inteiro de práticas acumuladas na biblioteca transversal da organização — incluindo uma camada de cache pensada para APIs de alto tráfego voltadas ao consumidor final. O serviço, que processava um lote de dados por hora, ganhou uma infraestrutura de cache que nunca teria uso real, enquanto as práticas que realmente importavam para um sistema financeiro — log de auditoria e garantia de idempotência em operações repetidas — ficaram diluídas no meio de um volume de código que não tinha relação com o domínio daquele serviço.

A biblioteca em si era boa. O problema foi tratá-la como um pacote a ser importado por inteiro, em vez de um catálogo do qual selecionar o que realmente se aplicava àquele contexto específico.

Biblioteca orienta escolha, não substitui julgamento

Uma biblioteca deve orientar escolhas, não ser copiada integralmente para todo projeto.

Uma biblioteca transversal — o repositório de padrões, convenções e soluções já validadas em outros projetos, mencionado no artigo sobre o AI Software Engineer Studio — existe para evitar que cada equipe reinvente as mesmas soluções para os mesmos problemas recorrentes. Mas nem toda prática documentada nessa biblioteca se aplica a todo projeto. Aplicar tudo indiscriminadamente cria o problema oposto ao que a biblioteca deveria resolver: complexidade desnecessária, importada por hábito, não por necessidade real do domínio.

Selecionar em vez de importar por inteiro

A pergunta correta não é "quais práticas da biblioteca existem", mas "quais características deste projeto específico determinam quais práticas se aplicam":

Selecionando práticas por características do projeto

Característica do projeto Prática da biblioteca aplicável
Lida com dados financeiros ou sensíveis Log de auditoria, idempotência, criptografia em repouso
Alto volume de requisições concorrentes de usuários finais Cache, rate limiting, circuit breaker
Processamento em lote, baixa concorrência Idempotência e retomada de falha, cache dispensável

Para a API de conciliação financeira do exemplo, a linha relevante é a primeira — e parcialmente a terceira, pela natureza em lote do processamento. A segunda linha, que motivou a camada de cache desnecessária, simplesmente não se aplicava ao perfil daquele serviço.

Quem decide o que se aplica

Delegar a um agente a leitura mecânica de "siga as diretrizes" sem contextualizar as características do projeto específico transfere para ele uma curadoria que deveria ser feita por quem entende o domínio. Um Context Pack bem escrito para essa tarefa deveria declarar explicitamente quais categorias de prática se aplicam — segurança e idempotência, por exemplo — e quais não, evitando que o agente decida sozinho importar tudo por precaução.

O limite: sem biblioteca nenhuma, o problema volta

A crítica à aplicação indiscriminada não é argumento contra ter uma biblioteca transversal. Sem ela, cada projeto volta a reinventar suas próprias práticas de log, autenticação e tratamento de erro, muitas vezes pior do que a versão já validada em outros lugares da organização. O problema nunca foi centralizar conhecimento — foi aplicá-lo sem considerar se ele realmente cabe no projeto em questão.

Conclusão

Uma biblioteca transversal é mais útil quando funciona como um catálogo consultável, do qual se seleciona o que o domínio exige, do que como um pacote a ser importado por inteiro em nome da consistência. Curadoria continua sendo trabalho humano — a biblioteca informa a escolha, não a substitui.
