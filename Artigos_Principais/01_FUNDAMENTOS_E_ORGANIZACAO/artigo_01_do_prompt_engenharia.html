Fundamentos e Organização Arco 1 — Da improvisação ao método

Do prompt à engenharia: como organizar o desenvolvimento de software com agentes de IA

Você abre o seu editor, aciona um agente de IA e pede: "Crie uma tela de login em React com Tailwind, consumindo uma API REST". Em poucos segundos, o código está na sua frente. Você copia, cola, roda o projeto e funciona. A sensação inicial é de que o trabalho de desenvolvimento acabou de ficar muito mais fácil.

Aí o projeto avança. No dia seguinte, você pede para o mesmo agente adicionar suporte a tokens JWT e adaptar aquele código para o Design System legado da sua empresa. De repente, o CSS quebra. Componentes somem. Pior: você percebe que o agente alterou as regras de roteamento que outra equipe tinha acabado de estabilizar.

O resultado é clássico: você passa as próximas três horas lendo e corrigindo o código que a IA gerou, tentando entender como ela bagunçou dependências que nem deveriam estar no escopo daquela tarefa. É a frustração de quem descobre que gerar um trecho de código isolado é fácil, mas construir software é outra história.

Gerar código virou commodity. Produzir software, não.

Engenharia de software orientada por IA não é sobre dominar técnicas de "engenharia de prompt". É sobre pegar a capacidade de geração de um modelo de linguagem e colocá-la dentro de um processo governado por contexto, escopo delimitado, registro de decisões e validação verificável.

A engenharia tradicional exige entender requisitos, alinhar integrações, garantir segurança e validar o que foi feito. Quando trazemos IA para a mesa, esses fundamentos não mudam em nada. O que muda é apenas o responsável por escrever as linhas iniciais da lógica.

O maior erro técnico que as equipes cometem hoje é tratar a memória de uma janela de chat como se fosse a documentação do projeto. Modelos como GPT-4 e Claude são ferramentas sem estado (stateless). Eles não lembram do que foi discutido na semana passada. Se as regras de negócio e a arquitetura do seu sistema vivem soltas num histórico de prompts, seu projeto já virou um código legado inavegável na primeira semana.

Para sair dessa improvisação, precisamos tratar o ambiente de desenvolvimento como um AI Software Engineer Studio. Isso significa parar de dar comandos genéricos e passar a trabalhar com três pilares inegociáveis:

  • Contexto Explícito: O agente não deve ter que vasculhar o repositório inteiro adivinhando padrões. Ele precisa receber as regras do domínio e a stack homologada especificamente para a tarefa em questão.
  • Escopo Fechado: O agente precisa saber quais arquivos ele tem autorização para alterar e quais pastas são bloqueadas (read-only). Escopo aqui é contenção de risco.
  • Rastreabilidade: Não basta a IA "dizer" que está pronto. Cada linha alterada precisa estar vinculada a uma Issue, respeitar um documento de decisão arquitetural (ADR) e passar nos testes automatizados.

O Pedido Solto vs. O Fluxo de Engenharia

Para entender a diferença na prática, vamos imaginar a criação de uma rota de autenticação para um SaaS.

O cenário improvisado (O Pedido Solto):
Você pede: "Crie uma API para autenticar usuários no nosso SaaS usando Node.js". A IA, otimizando pelo que é mais comum na internet, decide implementar isso em Express e salvar as credenciais num MongoDB, guardando as senhas em texto puro. Só que a arquitetura oficial da sua empresa usa Fastify e PostgreSQL, e senhas sem hash são uma falha crítica. O código funciona na teoria, mas é inútil — e perigoso — para a sua equipe.

O cenário de engenharia (Fluxo Governado):
A mesma funcionalidade é tratada como uma unidade de software real. O fluxo ocorre assim:

  1. Você anexa a Issue, que já define o contrato da API (exatamente o que entra no JSON de request e o que sai no response).
  2. Você referencia o ADR do projeto: "Decisão vigente: o banco de dados é sempre relacional (PostgreSQL) usando Prisma ORM".
  3. Você entrega um Context Pack (um pacote estruturado de contexto) ao agente. A instrução não é um parágrafo de texto livre, mas sim um conjunto claro de restrições de engenharia:
{
  "context_pack": {
    "domain": "Authentication",
    "allowed_stack": ["Fastify", "TypeScript", "Zod", "Prisma"],
    "forbidden_patterns": [
      "Senhas em texto puro são estritamente proibidas (use bcrypt)",
      "Não acesse o banco de dados direto na rota (use o padrão repository)"
    ],
    "quality_gates": ["unit_tests_must_pass", "eslint_no_warnings"]
  }
}

A partir do momento em que a IA recebe o contexto formalizado, ela deixa de ser um estagiário imprevisível e se torna um executor cirúrgico, restrito aos limites arquiteturais que a sua equipe decidiu.

Checklist de Delegação Segura para Agentes

  • A regra está fora do chat? A IA sabe o que construir com base numa Issue existente antes de começar a pensar em como codificar?
  • As restrições de tecnologia são explícitas? A stack e as bibliotecas proibidas foram declaradas na entrada?
  • Existe um contrato técnico mapeado? O input e output da funcionalidade estão alinhados com o resto do sistema?
  • Os critérios de aceite são executáveis? Como a IA vai fornecer a evidência (ex: logs de teste) de que a tarefa realmente terminou?

O trade-off: pagar o preço do setup

Em tecnologia, não existe almoço grátis. Exigir Context Packs, Issues formais e critérios de aprovação de código gera um atrito óbvio: o tempo de setup.

Para quem se acostumou com o atalho de copiar e colar prompts de uma linha no ChatGPT, formalizar regras de engenharia vai soar como burocracia desnecessária. O trade-off, no entanto, é matemático: você troca a ilusão do código instantâneo pela garantia da entrega real.

O tempo que um arquiteto ou desenvolvedor sênior investe estruturando as regras de contexto hoje é o tempo que toda a equipe não vai precisar gastar na semana que vem dando manutenção em uma arquitetura espaguete feita sem padrão nenhum. O rigor protege o futuro do produto.

Vale ressaltar que a governança deve ser proporcional ao problema. Você não precisa de um Context Pack complexo ou de um ADR aprovado para escrever um script isolado que converte um JSON no seu ambiente local. Mas se estamos falando de mexer em pagamentos, roteamento ou autenticação? O rigor técnico não é negociável.

A transição necessária

A passagem do uso casual de prompts para a orquestração de engenharia será a base de todos os próximos artigos desta série. Sem essa estrutura inicial, qualquer tentativa de escalar desenvolvimento usando agentes autônomos vai resultar apenas em geração de débito técnico em velocidade recorde.

Tokens de IA estão cada vez mais baratos. Geração de código virou uma ferramenta banal. Mas entregar software confiável, manutenível e seguro continua sendo um desafio focado em arquitetura e responsabilidade humana.

Resistir ao atalho e impor governança sobre o que as ferramentas de IA geram é o que define a linha entre quem apenas "sabe usar o Copilot" e quem atua como Engenheiro de Software.

Olhando para o seu repositório hoje: você está operando um sistema integrado de engenharia com agentes de IA, ou está apenas cruzando os dedos na hora de fazer o merge?
